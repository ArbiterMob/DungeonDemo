Progetto Modulo 1 - Fondamenti di Computer Graphics

DungeonDemo

1. Descrizione dell'applicazione

L'applicazione sviluppata consiste in una demo in terza persona, fortemente ispirata alle dinamiche del classico gioco arcade "Acchiappa la talpa", ma reinterpretata in chiave fantasy all'interno di un dungeon sotterraneo. Il software sfrutta le API WebGL per il rendering in tempo reale di geometrie complesse, l'applicazione di texture e la gestione dinamica dell'illuminazione e delle ombre.

Il giocatore controlla un fantasma (mainObject), il cui scopo è ripulire la stanza del dungeon eliminando tutti i ratti che si muovono liberamente e in modo imprevedibile sulla superficie di gioco. All'inizio della partita, tramite l'interazione con una cassa, il fantasma ottiene un martello, potendo così colpire i ratti.

L'atmosfera lugubre del dungeon è accentuata da elementi decorativi disposti nell'ambiente, tra cui scheletri e quadri appesi lungo le pareti. Una delle peculiarità scenografiche risiede nell'illuminazione: la sorgente luminosa principale non è statica, bensì è ancorata rigidamente al modello di una lampadina posizionata sotto un pipistrello che vola dinamicamente sopra l'area di gioco, proiettando coni di luce e ombre in continuo mutamento.

2. Spiegazione delle scelte effettuate

2.1 Architettura software e la classe SceneObject

Per garantire modularità, estensibilità e una netta separazione delle responsabilità, l'intera logica di gestione degli elementi grafici è stata incapsulata all'interno della classe SceneObject. Questa classe, utilizzando i file di utility glm_utils.js, load2_mesh.js e webgl-utils.js, astrae la complessità di basso livello di WebGL, occupandosi autonomamente di:

  • Caricamento di mesh da file in formato obj
  • Inizializzazione, caricamento e binding delle texture grafiche (Diffuse, Bump, Roughness).
  • Allocazione dei buffer di memoria della GPU (VBO) e popolamento sistematico di attributes e uniforms associati agli shader program attivi.
  • Gestione dello stato spaziale dell'oggetto tramite matrici di trasformazione affine (Traslazione, Rotazione).

2.2 Gestione delle Animazioni e Cinematica tramite Interpolazione Lineare

I movimenti degli oggetti non avvengono tramite variazioni discrete e istantanee, ma sono gestiti mediante tecniche di interpolazione lineare (LERP) calcolate frame per frame per garantire fluidità visiva indipendentemente dal frame rate. A tale scopo, sono stati implementati i seguenti algoritmi principali: 

// esempio di interpolazione lineare usato nel progetto
moveStraight(deltaTimePhysics) 
{
    // seguendo la logica del tempo della fisica, il movimento viene aggiornato ogni deltaTimePhysics 
    // rispetto a quanto si vuole farlo durare
    this.moveProgress += (deltaTimePhysics / moveDuration);

    // se il movimento è finito
    if (this.moveProgress >= 1.0)
    {
        this.moveProgress = 1.0;
        this.isMoving = false;
    }

    const p = this.moveProgress;

    // interpolazione lineare della posizione
    const currentX = this.startPosition[0] + (this.targetPosition[0] - this.startPosition[0]) * p;
    const currentZ = this.startPosition[2] + (this.targetPosition[2] - this.startPosition[2]) * p;

    this.position = [currentX, this.position[1], currentZ];
}
  • moveStraight(): Date le coordinate spaziali di partenza e di arrivo, la funzione calcola la posizione intermedia dell'oggetto a ogni frame tramite interpolazione lineare. Viene invocata per la traslazione fluida dei personaggi (ratti, pipistrello, bulbo, fantasma).
  • calculateWaypointStep(): Consente al pipistrello di muoversi lungo un percorso regolare predefinito. Una volta inizializzati i waypoint globali all'interno dell'oggetto, questa funzione determina dinamicamente la posizione successiva in direzione del waypoint di destinazione corrente.
  • rotatePivot(): Calcola l'angolo di rotazione incrementale attorno all'asse Z fino a un target predefinito, tenendo conto di un punto di pivot geometrico. Questo metodo è fondamentale per l'animazione di apertura della cassa (chestTop) e per il movimento di oscillazione del martello durante l'attacco.
  • updateRotation(): Gestisce la rotazione incrementale attorno all'asse verticale Y, permettendo agli oggetti in movimento di orientarsi fluidamente verso la direzione di marcia intrapresa ed evitando rotazioni istantanee o innaturali. L'algoritmo calcola la traiettoria angolare più breve (shortestAngle) per prevenire artefatti visivi dovuti allo scavalcamento dei 360 gradi: 
    updateRotation(deltaTimePhysics)
{
    this.rotationFaceProgress += (deltaTimePhysics / this.rotationFaceDuration);

    if (this.rotationFaceProgress >= 1.0)
    {
        this.rotationFaceProgress = 1.0;
    }

    let p = this.rotationFaceProgress;

    this.currentFacingAngle = this.startFacingAngle + this.shortestAngle * p;
}
    Data la posizione di destinazione (targetPos) e quella attuale, l'angolo di target viene ricavato trigonometricamente come segue: 
    setFacingDirection(targetPos, adjustAngle)
{
    this.startFacingAngle = this.currentFacingAngle;
    this.rotationFaceProgress = 0.0;

    let dx = targetPos[0] - this.position[0];
    let dz = targetPos[2] - this.position[2];
    this.targetFacingAngle = Math.atan2(dx, dz) + adjustAngle;

    let difference = this.targetFacingAngle - this.startFacingAngle;
    this.shortestAngle = Math.atan2(Math.sin(difference), Math.cos(difference));
}

2.3 Il Loop di Rendering

Il Render Loop è strutturato rigorosamente in tre fasi sequenziali, ottimizzando le performance e la sincronizzazione temporale:

  1. update(currentTime): Calcola il delta temporale rispetto al frame precedente (Time-based rendering). Aggiorna le logiche fisiche e le posizioni interpolate degli oggetti, slegando la velocità di gioco dalla frequenza di refresh dello schermo (FPS).
  2. render(): Questa funzione sfrutta una tecnica di rendering multi-pass: effettua prima la passata di generazione delle ombre (Shadow Pass) popolando la texture di profondità, e successivamente avvia la passata di rendering finale sulla scena visibile dall'utente.
  3. draw(): Funzione helper deputata all'invocazione dei comandi di disegno WebGL. Si occupa di inizializzare le variabili uniform comuni a tutti gli shader e di associare gli shader program specifici agli oggetti corrispondenti.

3.1 Logica di Movimento sulla Griglia e Rilevamento delle Collisioni

Lo stato strutturale del dungeon viene preservato in memoria attraverso due matrici bidimensionali distinte, deputate a mappare rispettivamente la disposizione degli elementi sul pavimento e sul soffitto. Il controllo spaziale è governato dalla funzione di validazione checkMoveFloor() presente nel file utils.js: prima di effettuare una traslazione, l'entità interroga la griglia e riceve un valore booleano (true se la casella di destinazione è libera da ostacoli, false in caso contrario).

  • Comportamento dei Ratti: Poiché il movimento dei ratti è randomico, qualora la casella selezionata risulti occupata (checkMoveFloor restituisce false), viene testata immediatamente un'altra direzione estratta casualmente. La selezione della direzione si basa sulla permutazione casuale di un array di indici direzionali mediante il seguente frammento logico: 
    let directions = [0, 1, 2, 3]; 
for (let j = directions.length - 1; j > 0; j--) 
{ 
    const k = Math.floor(Math.random() * (j + 1)); 
    [directions[j], directions[k]] = [directions[k], directions[j]]; 
}
  • Comportamento del mainObject: Per il fantasma guidato dall'utente, l'accesso a una casella occupata è vietato e questo genera contestualmente un feedback visivo sul log testuale dell'HUD a schermo, notificando l'impossibilità di procedere in quel punto dello spazio.
Particolarità del movimento del mainObject: Il movimento del fantasma non segue assi globali statici, ma si affida a un sistema di coordinate locali in cui i vettori di movimento (Avanti, Sinistra, Indietro, Destra) vengono calcolati a partire dall'angolo di rotazione corrente sul piano XZ. 
// Calcolo delle componenti vettoriali direzionali
const fx = Math.cos(mainObject.currentFacingAngle); 
const fz = - Math.sin(mainObject.currentFacingAngle); 

const lx = Math.cos(mainObject.currentFacingAngle + Math.PI / 2); 
const lz = - Math.sin(mainObject.currentFacingAngle + Math.PI / 2); 

const bx = - fx; 
const bz = - fz; 

const rx = Math.cos(mainObject.currentFacingAngle - Math.PI / 2); 
const rz = - Math.sin(mainObject.currentFacingAngle - Math.PI / 2);
Questo permette al fantasma di spostarsi con precisione nella direzione esatta verso cui il modello è orientato visivamente.

4. Particolarità tecniche

4.1 Modellazione 3D (Blender)

Tutti i modelli poligonali presenti nella scena (mainObject, hammer, chestTop, chestBot, rat, batCore, batLeftWing, batRightWing, lightBulb, skeleton, floor, pillar, ceiling e i vari paintings) sono stati realizzati utilizzando il software open-source Blender.

Per l'applicazione delle texture è stata eseguita una mappatura UV avanzata. È stata applicata la tecnica del Make Seam per definire i tagli strutturali sulle mesh e consentire uno sviluppo piano privo di distorsioni geometriche, coadiuvato dall'algoritmo Project From View (proiezione dei vertici UV correnti della mesh basata sulla vista prospettica 3D attiva).

4.2 Parsing del file materiali

Sono state aggiunte modifiche al parsing del file mtl nella funzione glmReadMTL del file glm_utils.js. Le proprietà aggiunte sono le seguenti:

  • map_Kd: Mappa la texture di base dell'oggetto tramite un'immagine.
  • map_Bump: Fornisce le coordinate delle normali espresse in Tangent Space.
  • map_Ns: Fornisce i coefficienti di roughness per ogni texel.

4.3 Architettura dei Fragment Shader

Per ottimizzare il carico computazionale sulla GPU, il rendering non utilizza un unico shader program monolitico, ma shader specifici che vengono usati selettivamente in base ai requisiti del materiale dell'oggetto:

  1. colorShader: Shader minimale di base utilizzato per lo shadow pass.
  2. lightShadowShader (Texture + Luce + Ombre): Applica la texture combinando il calcolo dell'illuminazione di Phong e l'attenuazione data dalla shadow map.
  3. lightShadowBumpShader (Texture + Luce + Ombre + Bump Mapping): Introduce il calcolo delle perturbazioni delle normali per pixel per simulare rilievi geometrici complessi.
  4. lightShadowBumpShininessShader (Texture + Luce + Ombre + Bump Mapping + Shininess Mapping): Integra tutte le caratteristiche precedenti, aggiungendo la modulazione della riflessione speculare (shininess) data dalla texture dedicata.

4.4 Gestione delle Ombre (Shadow Mapping)

La tecnica dello Shadow Mapping è implementata in modalità multi-pass. Nella prima passata (Shadow Pass), la scena viene renderizzata posizionando la camera in corrispondenza della sorgente luminosa proiettata da sotto alla lampada. Il risultato di questa passata viene memorizzato in una depthTexture.

Nella passata di rendering principale, all'interno del vertex shader, le coordinate di ciascun frammento vengono trasformate dallo spazio di vista del mondo allo spazio di vista della luce. Successivamente, nel fragment shader, effettuando un test di profondità, se il valore Z attuale del frammento risulta maggiore o uguale del rispettivo valore memorizzato nella depthTexture (coadiuvato da un valore di bias configurabile per prevenire l'effetto di shadow acne e peter panning), il frammento viene marcato come "in ombra" e privato della componente speculare e diffusa della luce diretta.

4.5 Bump Mapping

Per l'integrazione del Bump Mapping (implementato prendendo come riferimento metodologico la documentazione tecnica di LearnOpenGL), è stata sviluppata una funzione matematica (calculateTangent nel file utils.js) che analizza le facce triangolari delle mesh per calcolare le Tangenti (T) partendo dalle coordinate spaziali dei vertici e dalle coordinate UV associate.

Poiché le normali campionate dalla texture map_Bump risiedono nel tangent space, nel vertex shader viene costruita la matrice TBN (composta dai vettori Tangente, Bitangente e Normale). Moltiplicando il vettore campionato per la matrice TBN, le normali vengono convertite correttamente nel World Space, permettendo un calcolo coerente del prodotto scalare per l'illuminazione di Phong per-pixel.

4.6 Rimappatura della Roughness Map

I valori estratti dalla texture map_Ns derivano originariamente da mappe di roughness di tipo PBR, i cui valori fluttuano in un range normalizzato [0.0, 1.0]. Poiché il modello di illuminazione implementato si basa sull'equazione di riflessione di Phong, i valori estratti vengono rimappati nel range richiesto dal modello. Ciò consente di ottenere riflessi netti e focalizzati su materiali riflettenti (e.g. il martello) e riflessi morbidi e diffusi su superfici opache (e.g. il pavimento).

4.7 Configurazione dello Spotlight e Gestione della Penombra

Nei fragment shader dedicati alla gestione dei materiali illuminati viene modellata una sorgente di tipo Spotlight. I parametri geometrici descrittivi del cono luminoso vengono passati tramite le seguenti definizioni uniform e computati nel codice GLSL: 

// uniform nel codice javascript
u_innerLimit: Math.cos(Utils.degToRad(controlsLighting.fieldOfView / 2 - 10)), 
u_outerLimit: Math.cos(Utils.degToRad(controlsLighting.fieldOfView / 2)), 
u_lightDirection: lightWorldMatrix.slice(8, 11) // Corrisponde al vettore Z della matrice di trasformazione della luce
// Logica di attenuazione nel codice GLSL
float dotFromDirection = dot(surfaceToLightDirection, u_lightDirection); 
float limitRange = u_innerLimit - u_outerLimit; 
float inLight = clamp((dotFromDirection - u_outerLimit) / limitRange, 0.0, 1.0);

La logica matematica implementata permette di sfumare gradualmente gli ultimi 10 gradi del cono d'ombra. Le variabili e le operazioni si articolano secondo i seguenti criteri:

  • u_lightDirection esprime la direzione del vettore Z centrale della luce in world space. Eseguendo il prodotto scalare tra questo vettore e il vettore che unisce la superficie alla luce (surfaceToLightDirection), si ricava il coseno dell'angolo compreso tra l'asse di simmetria della sorgente luminosa e il punto della superficie considerata.
  • La uniform u_innerLimit definisce l'ampiezza del cono interno: qualsiasi pixel situato entro questo raggio angolare riceve il 100% dell'intensità luminosa. Al contrario, la uniform u_outerLimit delimita la soglia esterna del fascio, al di fuori della quale i frammenti non ricevono alcune luce.
  • La sottrazione dei coseni dei due limiti genera il limitRange, che quantifica l'estensione geometrica della penombra. La formula finale applicata alla variabile inLight opera un'interpolazione lineare inversa:
    • (dotFromDirection - u_outerLimit) misura lo scostamento angolare del pixel rispetto al confine del buio.
    • La divisione per limitRange normalizza tale scostamento in una percentuale compresa nell'intervallo [0.0, 1.0].
    • La funzione clamp(..., 0.0, 1.0) costringe tassativamente il coefficiente a non assumere valori negativi fuori dal cono e a non superare l'unità al centro del fuoco.
